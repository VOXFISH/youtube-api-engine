# Product Requirements Document
> **Project Rice**
>
> *유튜브 인터랙티브 게임을 위한 엔진*
## 1. 문서 개요

### 1.1 목적
> 본 문서는 OpenGL 기반 유튜브 인터랙티브 게임 엔진과
> 전용 방송 관리 애플리케이션의 요구사항을 정리하기 위한 요구분석서이다.
>
> 시스템은 게임 실행 파일, 전용 애플리케이션, YouTube 송출 기능, 데이터베이스 저장소를 포함하는 방송용 인터랙티브 게임 플랫폼을 목표로 한다.

### 1.2 시스템 개요
본 시스템은 다음 하위 시스템으로 구성된다.
- 게임 엔진
    | 하위 시스템 | 설명 |
    | --- | --- |
    | 게임 루프 | 게임 내부 함수를 실행과 렌더링을 반복 |
    | 렌더링 | 게임 오브젝트를 화면 또는 스트림의 형태로 출력 |
    | 오브젝트 | 게임 내 객체를 오브젝트 단위로 다룸 |
    | 씬 | 오브젝트와 컴포넌트를 ECS 구조로 저장 |
    | 스크립트 | 커스텀 스크립트 언어로 상용 Unity 엔진과 유사한 환경을 제공 |
    | 데이터 저장 | 내부 변수 및 DB 변수를 저장 |
- 전용 어플리케이션
    | 하위 시스템 | 설명 |
    | --- | --- |
    | 빌드 파일 관리 | 다수의 빌드 파일의 실행 및 스트리밍 여부를 확인 가능함 |
    | 데이터 저장 | Redis와 PostgreSQL을 이용해 임시/영구 데이터를 저장함 |
    | Youtube 연동 시스템 | 채팅 입력, 방송 제어, 스트리밍 데이터 송출을 담당함 |

### 1.3 주요 사용자 및 주체
본 명세서에서 시스템 사용 주체는 다음과 같이 구분되며, 서로 다른 요구사항을 가진다.
| 주체 | 설명 |
|---|---|
| 엔진 사용자 | 엔진을 이용해 인터랙티브 영상을 개발하는 개발자 |
| 방송 관리자 | 빌드파일과 전용 어플리케이션을 이용해 게임을 송출 및 운영하는 주체 |

## 2. 요약
### 2.1. 핵심 요구사항 요약
엔진은 Unity MonoBehavior와 비슷한 C++ 기반의 스크립팅 언어를 통해 유튜브 인터랙티브 게임의 실행 파일을 빌드할 수 있다. 스크립팅 언어는 상용 3D 게임 엔진이 제공하는 기본적인 기능을 제공하며, 전용 어플리케이션의 기능들을 조작할 수 있다.
빌드된 실행 파일은 전용 어플리케이션 없이 독립적으로 실행 가능하며, 이 경우 로컬 입력만을 지원한다. 전용 어플리케이션을 이용할 경우, 어플리케이션은 게임과 통신하여 게임 데이터 조작, Youtube 방송 제어, 화면 기반/데이터 기반 송출을 제공한다.

#### 3.2.5. 데이터의 저장

이 부분에서는 데이터의 분류 및 데이터베이스와 그 관리 관련 요구사항을 다룬다.

#### 3.2.5.1. 데이터 분류 체계

모든 저장 데이터는 다음 세 축으로 분류한다.

| 분류 축 | 값 | 설명 |
|---|---|---|
| 적용 범위 | bound / general | 단일 실행 인스턴스 전용 데이터인지 전체 스트리밍 공유 데이터인지 구분 |
| 보관 기간 | game / streaming / permanent | 게임 단위 임시 데이터, 방송 단위 임시 데이터, 영구 데이터 구분 |
| 접근 권한 | locked / open | 전용 애플리케이션과 게임 엔진의 읽기/쓰기 권한 구분 |
##### 3.2.5.1.1 bound / general

| 분류 | 의미 | 예시 | 저장 범위 |
|---|---|---|---|
| bound | 단일 실행 파일 또는 단일 게임 인스턴스에만 적용되는 데이터 | 현재 게임 점수, 현재 라운드 상태, 특정 실행 파일의 임시 변수 | 특정 게임 실행 인스턴스 |
| general | 전체 스트리밍 환경에서 공유되는 데이터 | 블랙리스트, 게임 시작 일자, 후원 정보, 공통 설정 | 전체 애플리케이션/스트리밍 세션 |
##### 3.2.5.1.2. game / streaming / permanent

| 분류 | 의미 | 저장소 | 삭제 시점 |
|---|---|---|---|
| game | 게임 실행 중에만 유지되는 데이터 | Redis | 게임 종료 시 삭제 |
| streaming | 스트리밍 세션 동안 유지되는 데이터 | Redis | 스트리밍 종료 시 삭제 |
| permanent | 영구 저장 데이터 | PostgreSQL | 명시적 삭제 전까지 유지 |

##### 3.2.5.1.3. locked / open

| 분류 | 의미 | 애플리케이션 권한 | 게임 엔진 권한 | 예시 |
|---|---|---|---|---|
| locked | 게임 엔진 또는 게임 스크립트가 소유하는 데이터 | 읽기 가능, 쓰기 불가 | 읽기/쓰기 가능 | 현재 체력, 내부 판정값, 게임 로직 상태 |
| open | 전용 애플리케이션이 조작할 수 있는 데이터 | 읽기/쓰기 가능 | 읽기 가능 | 게임 시작 여부, 보정값, 방송 관리자 명령, 이벤트 트리거 |

##### 3.2.5.1.4. 데이터 키워드 조합 예시

| 데이터 예시 | 범위 | 수명 | 권한 | 저장소 | 설명 |
|---|---|---|---|---|---|
| 현재 라운드 점수 | bound | game | locked | Redis | 특정 게임 실행 중에만 유지되며 게임 로직이 갱신 |
| 현재 이벤트 상태 | bound | game | locked | Redis | 단일 게임 인스턴스 내부 상태 |
| 방송 중 누적 참여자 수 | general | streaming | locked | Redis | 방송 중 전체 게임에서 공유되는 참여 통계 |
| 방송 관리자 이벤트 트리거 | general | streaming | open | Redis | 앱에서 수정하고 게임이 읽는 방송 제어 데이터 |
| 게임 시작 일자 | general | permanent | open | PostgreSQL | 앱에서 관리하고 여러 방송에서 참조 가능 |
| 블랙리스트 | general | permanent | open | PostgreSQL | 앱에서 등록/수정하고 게임이 조회 |
| 후원 금액별 리액션 정보 | general | permanent | open | PostgreSQL | 방송 전체에서 공유되는 영구 데이터 |
